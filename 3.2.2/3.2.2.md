# 3.2.2 手写数字识别系统交互流程设计

## 1. 系统概述

### 1.1 系统功能描述

本系统基于 MNIST 手写数字识别模型（`mnist.onnx`），实现对用户上传的手写数字图片进行自动识别，并返回识别结果（数字 0–9）。系统的核心目标是：

- 接收用户输入的手写数字图片（如 `img_test.png`）；
- 对图片进行标准化预处理，使其符合模型输入要求；
- 调用卷积神经网络（CNN）模型完成推理；
- 将预测结果以直观、友好的方式呈现给用户。

### 1.2 技术架构

系统采用基于 ONNX Runtime 的推理流程，完整管道如下：

```
加载模型（mnist.onnx）
    ↓
加载手写图片（img_test.png）
    ↓
预处理
  ├─ 灰度化（转为 'L' 模式）
  ├─ 缩放至 28×28 像素
  ├─ 转为 float32 数组
  ├─ 添加批次维度（axis=0）→ shape (1, 28, 28)
  └─ 添加通道维度（axis=0）→ shape (1, 1, 28, 28)
    ↓
模型推理（onnxruntime.InferenceSession）
    ↓
输出识别数字（argmax 取最大概率类别）
```

**核心技术组件：**

| 组件 | 说明 |
|------|------|
| 模型文件 | `mnist.onnx`（预训练卷积神经网络） |
| 推理引擎 | `onnxruntime.InferenceSession` |
| 图像处理 | PIL（`Image`）、NumPy |
| 输出标签 | 数字类别 0–9 |

## 2. 最优人机交互方式设计

### 2.1 交互界面设计

综合考虑易用性、部署便捷性与扩展性，推荐采用 **Web 界面**作为主要交互方式，辅以 **API 接口**支持程序化调用。

**推荐方案：Web 图形化交互界面**

- 前端：HTML + CSS + JavaScript（可选 React/Vue 框架）
- 后端：Python Flask / FastAPI
- 推理引擎：onnxruntime（服务端运行）
- 部署形式：本地服务或云端服务

**备选方案对比：**

| 方案 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| Web 界面 | 通用用户、演示场景 | 无需安装、跨平台、直观 | 需要网络环境 |
| 桌面应用 | 离线环境、本地使用 | 离线可用、响应快 | 需安装部署 |
| API 接口 | 开发者、系统集成 | 灵活、可批量调用 | 需编程能力 |
| 命令行工具 | 批量测试、脚本集成 | 轻量、自动化 | 用户体验较差 |

### 2.2 用户操作流程

#### Step 1：用户上传图片

**操作方式：**
- 用户进入 Web 界面，通过"点击上传"按钮或拖拽方式提交手写数字图片。
- 系统支持常见图片格式：`.png`、`.jpg`、`.jpeg`、`.bmp`。
- 上传后，系统在界面左侧显示原始图片预览，供用户确认内容。

**界面提示：**
- 上传区域显示提示文字："请上传包含单个手写数字（0-9）的图片"。
- 图片大小限制建议不超过 5MB，超出时给出提示。

---

#### Step 2：系统自动预处理

用户点击"开始识别"按钮后，后端自动执行以下预处理步骤（对用户透明）：

```python
from PIL import Image
import numpy as np

# 步骤 1：打开图片并转为灰度图
image = Image.open("img_test.png").convert('L')

# 步骤 2：缩放至 28×28
image = image.resize((28, 28))

# 步骤 3：转为 float32 数组
image_array = np.array(image, dtype=np.float32)

# 步骤 4：归一化（将像素值缩放至 [0, 1]）
image_array = image_array / 255.0

# 步骤 5：添加批次和通道维度，最终 shape 为 (1, 1, 28, 28)
image_array = np.expand_dims(image_array, axis=0)
image_array = np.expand_dims(image_array, axis=0)
```

界面显示"正在处理中..."的加载动画，提升用户等待体验。

---

#### Step 3：模型推理与结果展示

**推理过程：**

```python
import onnxruntime as ort

# 加载模型
session = ort.InferenceSession("mnist.onnx")

# 构建输入字典
input_name = session.get_inputs()[0].name
ort_inputs = {input_name: image_array}

# 执行推理
ort_outs = session.run(None, ort_inputs)

# 提取预测结果
predicted_digit = int(np.argmax(ort_outs[0]))
```

**结果展示方式：**

界面右侧展示以下内容：
1. **识别结果**：以大号字体醒目展示识别出的数字（如"识别结果：**7**"）。
2. **置信度分布**：以柱状图形式展示模型对 0–9 各类别的概率分布，帮助用户了解模型置信程度。
3. **预处理图片预览**：展示经过缩放和灰度化后的 28×28 图片，便于用户理解预处理效果。

---

#### Step 4：结果反馈与优化

提供用户反馈机制，持续改善系统性能：

- **结果确认**：界面提供"识别正确"与"识别错误"两个按钮，用户可点击反馈。
- **手动纠错**：若识别错误，用户可从下拉框中选择正确数字（0–9），提交纠错信息。
- **历史记录**：界面下方展示本次会话中的识别历史（图片缩略图 + 识别结果）。
- **重新识别**：用户可点击"重新上传"按钮，清空当前结果，上传新图片。

### 2.4 错误处理与用户提示

| 错误类型 | 触发条件 | 用户提示内容 |
|----------|----------|--------------|
| 文件格式错误 | 上传非图片文件 | "请上传有效的图片文件（支持 PNG、JPG、BMP）" |
| 文件过大 | 图片超过 5MB | "图片大小超出限制，请压缩后重新上传" |

| 模型加载失败 | `mnist.onnx` 文件缺失或损坏 | "模型加载失败，请联系管理员" |
| 推理超时 | 服务端处理时间过长 | "识别超时，请稍后重试" |
| 服务器异常 | 后端服务不可用 | "服务暂时不可用，请稍后重试" |

所有错误提示均以非技术性语言呈现，避免暴露底层异常信息。

## 3. 交互优化建议

### 3.1 性能优化

**批量处理：**
- 支持用户一次上传多张图片（建议上限 20 张），后端使用批量推理：

```python
# 将多张图片堆叠为 (N, 1, 28, 28) 的批次输入
batch_input = np.stack([preprocess(img) for img in images], axis=0)
ort_inputs = {input_name: batch_input}
ort_outs = session.run(None, ort_inputs)
predicted_digits = ort_outs[0].argmax(axis=1)
```

**缓存机制：**
- 对相同 MD5 哈希值的图片缓存识别结果，避免重复推理。
- 模型在服务启动时一次性加载（`InferenceSession` 全局复用），避免每次请求重新加载。

```python
# 服务启动时加载一次模型
session = ort.InferenceSession("mnist.onnx")

# 后续所有请求复用该 session
```

**异步处理：**
- 对于批量识别任务，采用异步任务队列（如 Celery + Redis），前端轮询结果，避免请求超时。

---

### 3.2 用户体验优化

- **实时预览**：图片上传后立即显示预览，无需等待识别完成。
- **进度反馈**：识别过程显示进度条或加载动画（预计耗时 < 1 秒）。
- **响应式设计**：界面适配桌面端与移动端，支持手机拍照后直接上传。
- **手写画板**：提供在线手写画板，用户可直接在网页上书写数字，无需上传图片文件，降低使用门槛。
- **结果复制**：识别结果支持一键复制到剪贴板。
- **多语言支持**：界面文字支持中英文切换。

---

### 3.3 可靠性保障

- **输入校验**：后端对上传图片进行严格的格式、大小、内容校验，防止恶意输入。
- **异常捕获**：推理过程使用 `try-except` 块捕获所有异常，确保服务稳定运行。
- **日志记录**：记录每次识别请求的时间戳、图片哈希、识别结果及耗时，便于问题排查。
- **健康检查**：提供 `/health` 接口，供监控系统定期检测服务状态。
- **模型版本管理**：模型文件路径和版本号通过配置文件管理，支持热替换。

## 4. 实施建议

### 4.1 技术选型建议

| 层级 | 推荐技术 | 说明 |
|------|----------|------|
| 前端 | HTML5 + JavaScript（Fetch API） | 轻量、无框架依赖，易于维护 |
| 后端 | Python FastAPI | 异步支持好、性能高、自动生成 API 文档 |
| 推理引擎 | onnxruntime 1.x | 官方支持、跨平台、推理速度快 |
| 图像处理 | Pillow + NumPy | 成熟稳定、与 onnxruntime 兼容性好 |
| 部署 | Docker 容器化 | 环境一致、便于迁移和扩展 |

### 4.2 开发优先级建议

1. **第一阶段**：实现核心推理 API（单张图片上传 → 识别结果返回）；
2. **第二阶段**：开发 Web 界面，集成图片上传、结果展示、置信度可视化；
3. **第三阶段**：增加在线手写画板、批量识别、历史记录等扩展功能；
4. **第四阶段**：完善错误处理、日志监控、Docker 部署方案。

### 4.3 测试建议

- 使用标准 MNIST 测试集图片验证系统端到端识别准确率；
- 测试边界情况：空白图片、非数字图片、低分辨率图片、旋转图片；
- 进行并发压力测试，确保多用户同时使用时服务稳定；
- 在移动端设备上验证界面响应式布局的显示效果。

## 总结

本方案以 **Web 图形化界面 + REST API** 为核心，围绕 `mnist.onnx` 卷积神经网络模型，设计了一套完整的手写数字识别人机交互流程。交互设计遵循"简单上传 → 自动处理 → 直观展示 → 友好反馈"的四步原则，兼顾普通用户的易用性与开发者的扩展性。通过模型预加载、批量推理、结果缓存等优化手段，系统可在 1 秒内完成单张图片的识别，满足实时交互需求。
