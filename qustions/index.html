1.2.4 智能卖点生成系统业务模块效果优化

题号 1.2.4-1

最强烈的问题 + 原因（考场精简版）

核心采分点： 主打题干给的两个：卖点生成不准确 + 缺乏个性化定制。原因落到卖点这个产品的命脉——「用不了」和「没差异」：不准→不能体现真实卖点甚至虚假宣传；不个性化→千篇一律、没品牌调性。

问题一：卖点生成不准确

系统生成的卖点与产品真实属性不符——遗漏核心卖点、套话空泛，甚至出现夸大或与产品无关的"幻觉"信息。用户不满的原因：

  • 卖点是营销转化的核心，生成不准就无法体现产品真实优势，用户（商家/运营）不能直接使用，必须逐条人工核对修改，违背了「用 AI 提效」的初衷，效率不升反降；
  • 若错误卖点直接发布，会误导消费者、引发投诉，甚至构成虚假宣传风险，损害品牌信任。

问题二：缺乏个性化定制

系统输出千篇一律，无法根据不同品牌调性、目标人群、产品品类、投放平台和营销场景调整。用户不满的原因：

  • 不同品牌/品类/平台对卖点的风格要求差异极大（高端品牌 vs 性价比、小红书种草 vs 天猫详情页），通用化输出缺乏差异化和品牌调性，拿来即用性差；
  • 用户无法按需求（语气、长度、卖点角度、目标人群）定制，体验僵硬、缺乏掌控感，难以形成依赖和粘性。

一句话归纳： 不准确让卖点「不能用」，缺乏个性化让卖点「不好用」，两者都直接击穿了智能卖点生成系统的核心价值。

题号 1.2.4-2

方案 + 步骤 + 效果（考场精简版）

核心采分点： 六步骨架；准确率靠产品知识库 + RAG 检索增强 + 领域微调 + 事实校验，个性化靠用户/品牌画像 + 多维参数化定制。

一、关键实施步骤（六步）

  1. 问题诊断： 做 badcase 分析，定位卖点不准的根因（产品信息缺失、模型幻觉、还是意图理解偏差）；梳理用户的定制化需求（品牌调性、受众、平台、风格）。
  2. 数据层面： 构建并完善高质量产品信息知识库（规格、参数、真实卖点、品类术语）；收集分品类/品牌/风格标注的优质卖点文案语料；针对 badcase 补充难例。
  3. 模型层面（提准确率）： 引入 RAG 检索增强——生成时检索真实产品信息与知识库，让卖点"有据可依"，大幅减少幻觉、保证与产品一致；用领域语料微调大模型提升文案质量；加入事实校验/属性约束，确保卖点基于真实产品属性。
  4. 个性化层面（补短板）： 支持多维参数化定制（品牌调性、目标人群、投放平台、文案长度、卖点角度——功能/情感/场景）；建立品牌库与用户画像，按品牌风格生成；提供风格模板与指令定制入口。
  5. 测试上线： 评估卖点与产品的一致性（准确率）、可直接采用率、用户满意度；A/B 测试 + 灰度发布。
  6. 持续迭代： 建立反馈闭环，收集用户的采纳/修改数据持续优化（可结合人工反馈微调 RLHF），并随新品上市更新知识库。

二、预期效果

卖点生成准确率显著提升、与产品真实属性一致、幻觉大幅减少，可直接采用率提高；支持品牌/受众/平台/风格的个性化定制，输出有差异化和品牌调性；减少人工修改成本，最终用户满意度、采纳率与粘性全面提升，系统服务质量增强。

  3. 傍晚（18—20 时）Relax Mode 使用次数全天最高，达 65 次：这是 Relax Mode 的黄金时段，用户在结束一天工作后需要放松，系统应在此时段主动推荐或自动切换至 Relax Mode。

2.3 场景使用规律总结

  • 频繁使用（推荐重点优化）：Relax Mode（全天均衡+傍晚高峰）、Reading Mode（白天主力场景）
  • 使用适中：Work Mode（早晨和上午集中使用）

题号 3.1.2-2：智能照明系统优化方向与解决方案

基于上述分析，提出以下三个核心优化方向：

优化方向一：构建时段自适应照明推荐引擎，提升用户使用体验

问题背景：

数据显示用户的亮度与色温偏好具有显著的时间规律性：早晨偏好高亮度（均值 51.7）、夜晚偏好低亮度暖光（22 时均值仅 43.9，色温降至 3,339 K）；傍晚（18—20 时）Relax Mode 使用次数全天最高（65 次），夜晚 Sleep Mode 成为主导场景。然而目前这些场景均需用户手动触发，缺乏智能预判与主动推送机制。

解决方案：

  1. 建立用户照明行为模型：利用历史数据挖掘每位用户在不同时段的亮度、色温与场景偏好，为每个用户生成个性化的"照明时刻表"，实现"千人千面"的照明策略。

  2. 开发时段自动切换功能：根据时间规律，在关键节点（如 18:00 自动切换至 Relax Mode、21:00 自动切换至 Sleep Mode 准备状态）向用户推送建议或自动执行切换，减少手动操作频次。

  3. 引入渐进式亮度调节：夜间逐步降低亮度（如每 30 分钟降低 5 个单位）并自动收缩色温至暖光区间（< 3,500 K），模拟自然光变化规律，辅助用户建立健康的昼夜节律。

  4. 拓展 Sleep Mode 的午休场景：数据显示 Sleep Mode 在日间使用率极低，可推出"午休模式"子场景（13:00—14:00 时段），结合轻柔降亮与暖光设置，挖掘 Sleep Mode 在非夜间时段的使用价值。

预期效果： 减少用户手动操作次数约 30%—40%，提升场景触发的及时性与准确率，增强系统"懂用户"的智能感知。

优化方向二：针对高延迟场景实施分级优化，将系统响应时间压缩至 1.0 秒以内

问题背景：

当前系统平均响应时间为 1.064 秒，P90 达 1.804 秒，P99 接近 2.0 秒，27.5% 的请求响应超过 1.5 秒，用户体验存在明显延迟感。Relax Mode 是高延迟次数最多的场景（36 次高延迟），上午（10—12 时）是响应时间最长的时段（均值 1.125 s）。智能照明的核心体验是"即指即变"，超过 1 秒的响应延迟会显著降低用户满意度。

解决方案：

  1. 优化 Relax Mode 光效算法：对 Relax Mode 的场景渲染逻辑进行代码层优化，采用异步预加载机制，在用户触发指令前预先完成色温过渡参数计算，将 Relax Mode 平均响应时间目标压缩至 0.8 秒以内。

  2. 部署本地边缘计算网关：在家庭局域网内部署照明控制边缘节点，将常用指令（场景切换、亮度调节）的处理逻辑下沉至本地执行，避免每次操作都依赖云端往返通信，可将响应时间减少 40%—60%。

  3. 引入指令预测与预执行机制：结合用户行为规律（如傍晚大概率切换 Relax Mode），在预测窗口期（切换前 10—15 分钟）预热相关设备参数，实现"零感知延迟"的场景切换。

  4. 建立响应时间分级 SLA 监控机制：设置实时告警阈值（P50 < 0.8s，P90 < 1.5s，P99 < 1.8s），对持续超标的设备或时段触发自动诊断和降级保护策略，防止长尾延迟扩散。

  5. 高峰时段动态扩容：针对上午（10—12 时）的并发高峰，实施弹性资源调度策略，在预测到高并发前提前扩展服务处理能力，平抑峰值期响应时间波动。

预期效果： 将整体平均响应时间从 1.064 秒降低至 0.8 秒以下，P90 响应时间从 1.804 秒降低至 1.2 秒以内，高延迟（>1.5s）请求占比从 27.5% 降低至 10% 以下。

优化方向三：强化多场景智能联动与个性化推荐，提升场景使用均衡度

问题背景：

数据显示四大场景使用频率存在分化：Relax Mode（27.3%）和 Reading Mode（26.4%）占据主导，而 Work Mode（23.7%）和 Sleep Mode（22.6%）使用相对较少。Sleep Mode 在非夜间时段几乎处于"闲置"状态，场景价值未被充分挖掘。此外，各场景之间缺乏智能联动逻辑，用户需要频繁手动切换，降低了系统的智能化体验。

解决方案：

  1. 构建场景联动逻辑链：基于数据发现的"Work Mode（早晨）→ Reading Mode（白天）→ Relax Mode（傍晚）→ Sleep Mode（夜晚）"自然使用链条，设计场景自动流转规则，支持用户配置个性化的"一天照明计划"，系统按计划自动执行场景切换。

  2. 引入多设备联动协同：将智能照明与其他智能家居设备（如智能窗帘、空调、音乐）进行场景联动，例如触发 Relax Mode 时同步降低窗帘透光率并播放轻音乐，打造沉浸式场景体验，提升场景黏性和使用频率。

  3. 开发使用频率可视化反馈面板：在 APP 中为用户展示各场景的使用统计（频率、时长、亮度/色温分布），结合健康照明建议（如"您今天夜晚使用高色温照明超过 2 小时，建议切换至暖光模式"），引导用户优化照明习惯，间接提升低频场景的使用率。

  4. 针对 Work Mode 和 Sleep Mode 进行场景细分： - Work Mode 可细分为"高强度专注（高亮度 + 高色温）"与"轻度工作（中亮度 + 中色温）"两档，满足不同工作强度的需求； - Sleep Mode 增加"午休模式"与"深度睡眠模式"，前者适用于 13:00—14:00 短暂休息，后者在 22:00 后自动触发并逐步关闭全部灯光，丰富场景覆盖面。

  5. 建立个性化场景推荐算法：利用协同过滤算法，基于相似用户的行为模式向当前用户推荐其尚未使用或使用较少的场景组合，逐步提升系统的场景多样性使用率，使四大场景的使用频率趋于更均衡的分布（目标：各场景偏差不超过 3%）。

预期效果： 提升场景切换的智能化程度，减少用户手动干预；Sleep Mode 日间使用率提升 50% 以上；四大场景使用均衡度显著改善，用户整体满意度提升。

题号 3.1.3-2 智能健康手环产品优化方向与解决方案

基于上述数据分析，结合用户活动模式、健康指标关注度和数据同步性能三个维度的发现，提出以下 3 个优化方向及对应解决方案：

优化方向一：降低心率数据同步延迟，提升实时监测体验

问题背景：

心率数据的平均同步延迟为 1.502 秒，是步数和睡眠时长（约 0.30 秒）的 5 倍，且延迟范围集中在 1.0 - 2.0 秒，无法满足用户在运动高峰时段（06:00 - 08:00、17:00 - 20:00）实时查看心率的需求。在晨练或傍晚运动期间，超过 1.5 秒的延迟会显著降低用户的即时反馈体验。

解决方案：

  1. 优化心率信号处理算法： 采用轻量级的光电容积描记法（PPG）实时算法，将心率计算从云端或手机端迁移至手环本地芯片（边缘计算），减少数据上传后再处理的环节，预计将心率延迟压缩至 0.5 秒以内。

  2. 引入差分传输机制： 针对心率连续数据，改用增量传输代替全量传输——仅上报心率变化量而非完整波形数据，降低单次传输的数据包体积，减少蓝牙传输占用时间。

  3. 建立心率数据优先级通道： 在手环与手机应用的蓝牙通信协议中，为心率数据设置高优先级传输通道，确保在步数、睡眠等数据同步过程中心率数据不被阻塞排队。

  4. 设置心率预缓存机制： 在用户查看心率前 0.5 秒预测查看行为（如用户手腕抬起动作触发），提前完成心率数据同步，实现"准实时"显示效果。

预期效果： 心率同步延迟从平均 1.502 秒降低至 0.4 秒以内，与步数/睡眠时长延迟趋于一致，整体平均延迟从 0.698 秒降低至约 0.35 秒。

优化方向二：基于活动双峰规律，推出智能运动提醒与目标激励功能

问题背景：

用户活动呈现明显的"双峰"规律：早高峰（06:00 - 08:00，约 5,068 步/小时）和晚高峰（17:00 - 20:00，约 5,007 步/小时），而工作/学习期间（09:00 - 16:00）活动长期处于低谷（约 1,085 - 1,125 步/小时）。低谷时段步数仅为高峰的 21%，存在久坐风险。此外，各工作日与周末活动差异仅约 2.2%，用户行为模式固化，缺乏动态激励机制。

解决方案：

  1. 推出"午间久坐提醒"功能： 根据数据显示午间（13:00）是全天活动最低谷，在连续静止超过 45 分钟后发送个性化振动提醒，提示用户起身活动，提供简短的拉伸或步行建议（如"已久坐 60 分钟，建议起身走 100 步"）。

  2. 构建个性化双峰目标体系： 依据用户历史活动数据，分别为早高峰和晚高峰设置独立的步数目标（如"早晨目标：5,000 步，当前完成 78%"），利用用户主动参与高峰活动的习惯，强化目标完成感和正向反馈循环。

  3. 增加"工作日活跃度积分"机制： 针对工作日（周一至周五）日间低谷时段额外增加积分权重，鼓励用户在非高峰时段增加活动量。每完成一次日间步行（如 500 步/次）即触发积分奖励，通过游戏化设计提升低谷时段的活动意愿。

  4. 开发"AI 运动节奏建议"功能： 基于 7 天活动模式学习，预测当日活动轨迹，在用户活动量低于历史均值 20% 时主动推送运动建议，帮助用户维持规律运动习惯。

预期效果： 日间低谷时段（09:00 - 16:00）平均步数提升 15% - 25%，久坐时长减少，用户日活跃度和功能使用黏性显著提升。

优化方向三：深化三大健康指标的数据整合与综合健康洞察能力

问题背景：

数据显示用户对步数（33.92%）、心率（33.08%）、睡眠时长（33.00%）三项指标的关注度高度均衡，表明用户已形成全面监测健康的需求，而非单纯依赖某一指标。然而，当前产品仅提供单项指标的独立展示，缺乏跨指标的关联分析与综合健康评估，无法满足用户"了解整体健康状况"的深层需求。

解决方案：

  1. 推出"每日健康综合评分"功能： 将步数完成率、日间平均心率、夜间睡眠时长三项指标加权合成为每日"健康综合指数"（0 - 100 分），在手环屏幕和 App 首页以直观仪表盘展示，帮助用户一眼了解当天整体健康状态，减少多次分别查看各指标的操作。

  2. 构建"活动-心率-睡眠"关联分析报告： 基于历史数据挖掘三项指标的内在关联（如运动量与睡眠质量的关系、心率变异性与疲劳程度的对应），每周生成"健康洞察报告"，为用户提供数据驱动的个性化建议（如"本周运动量充足，但睡眠时长低于推荐值，建议提前 30 分钟就寝"）。

  3. 增加"异常指标预警"功能： 利用三项指标的历史基线数据，当某项指标出现显著偏离时（如心率异常升高或睡眠时长骤降）自动触发预警通知，提升手环的健康守护价值，增强用户对产品的依赖度与信任感。

  4. 优化多指标同屏展示体验： 在手环 App 中设计"健康概览"页面，将步数进度、心率趋势、睡眠时长三项指标整合至单一可滑动卡片，减少用户在各指标页面间的跳转次数，提升查看效率。鉴于三项指标查看频次均衡，一体化展示设计将显著提升操作便捷性。

预期效果： 用户单次开启 App 的信息获取效率提升 40% 以上，综合健康评分功能激活用户每日主动查看习惯，用户留存率和 30 天活跃率提升明显。

题号 3.1.4-2 智能健康监测系统优化方向与解决方案

优化方向一：构建智能化个性化预警机制，降低误报率

问题背景：

当前系统对餐后血糖（08:00、13:00、19:00 均值6.8—6.9 mmol/L）和晨峰血压（07:00 均值131.4 mmHg）的预警可能因阈值固化而产生误报。血糖在餐后1小时接近7.8 mmol/L临界值，传感器微小误差即可导致频繁误判；血压偏高发生率在晨起时段高达50%，若使用统一阈值，将对大量用户触发不必要的预警。

解决方案：

  1. 建立个体化动态基线模型：系统收集用户7—14天的历史数据，为每位用户建立个性化的"正常波动范围"基线，替代通用阈值。例如针对餐后血糖，以该用户历史餐后均值±1个标准差为预警参考范围，而非采用统一的7.8 mmol/L。

  2. 引入时间感知型分级预警：根据本报告识别的高风险时段（07:00晨峰、08:00/13:00/19:00餐后），设置"时段敏感性权重"。高风险时段适当降低预警触发阈值，以捕捉真实风险；相对安全时段（如18:00晚餐前血糖低谷期）适当提高容忍阈值，减少无效打扰。

  3. 多指标联合验证机制：单一指标触发预警前，先结合同时段其他指标进行交叉验证。例如，血糖偏高的同时若心率、血压均正常，则降级为"提示"而非"警报"，减少用户的预警疲劳。

  4. 传感器误差补偿算法：针对血糖检测传感器，引入温度、湿度等环境因子的误差补偿模型，将血糖测量误差控制在±0.05 mmol/L以内，从源头减少因测量误差导致的临界值误判。

预期效果： 降低误报率40%以上，提高预警的用户接受度和有效响应率。

优化方向二：系统架构优化，解决高峰时段响应延迟问题

问题背景：

分析显示午间（12:00—14:00）平均响应时间0.667秒为全天最高，慢响应（>0.8秒）次数高达8次；清晨（06:00—09:00）慢响应5次，且与血压晨峰高风险时段重叠，延迟预警的危害尤为突出。总体而言，19.0%的监测记录响应时间超过0.8秒，系统实时性有待提升。

解决方案：

  1. 实施错峰调度策略：针对07:00时段同时触发血压、血糖、体脂三项监测的并发冲突，设计智能调度引擎，将体脂分析任务（非实时急迫性指标）自动错峰至07:05或07:10执行，优先保障血压和血糖的实时响应。

  2. 引入边缘计算架构：将基础的健康指标计算（如血糖值换算、血压分级判断）从云端迁移至终端设备本地进行，减少网络传输延迟。本地计算可将响应时间目标压缩至0.3秒以内，仅将需要复杂分析的数据上传云端。

  3. 建立监测任务优先级队列：按健康风险等级为监测任务分级： - P0级（立即执行，响应<0.3s）：血压异常、血糖严重偏高的实时预警 - P1级（优先执行，响应<0.5s）：常规血压监测、餐后血糖检测 - P2级（后台执行，响应<1.0s）：体脂分析、历史数据同步

  4. 预测性缓存机制：基于用户历史监测时间规律，在高频监测时段（如07:00、12:00、19:00）前30秒预先加载相关计算资源和用户基线数据，减少冷启动延迟。

预期效果： 将高峰时段平均响应时间从0.667秒降至0.4秒以内，慢响应（>0.8秒）发生率从19.0%降至5%以下。

优化方向三：提升体脂分析功能的用户价值与使用体验

问题背景：

体脂分析仅占总监测记录的4.8%，且8天内体脂率数值恒为0.17，毫无变化，反映出两大核心问题：一是体脂分析的用户感知价值极低（数据无变化等于无反馈），二是可能存在传感器精度不足或测量方法固化导致的数据伪稳定问题。如不加以改进，该功能将逐渐被用户遗弃。

解决方案：

  1. 优化测量精度与方法： - 引入多维度体成分分析算法，结合用户身高、体重、年龄、性别等基础信息，采用生物电阻抗（BIA）多频率测量，提高体脂率的测量灵敏度，确保体脂率变化超过0.1%即可被检测到。 - 增加肌肉量、内脏脂肪、骨量等维度，将单一"体脂率"升级为"身体成分全景报告"，提升数据信息密度。

  2. 调整监测策略并强化趋势反馈： - 将体脂监测频率从每日1次调整为每周固定日期（如每周一07:00）进行，符合体脂变化的实际生理周期（以周为单位），消除日度"无变化"带来的无效感。 - 建立周度/月度趋势图，即使单次测量变化微小，通过可视化趋势曲线让用户直观感受到健康管理的长期效果（如"过去30天体脂率下降0.5%"），强化正向激励。

  3. 构建体脂与其他指标的联动分析： - 将体脂率与血压、血糖数据进行关联分析，自动生成健康风险评估报告。例如："您的体脂率0.17处于偏低区间，结合近期血压偏高趋势，建议关注心血管健康"，将孤立的体脂数据赋予行动指引价值。 - 根据体脂水平为用户提供个性化的运动和饮食建议，将被动监测转变为主动健康管理工具，显著提升该功能的用户留存率。

  4. 引入目标设定与成就激励机制： - 允许用户设定体脂目标（如3个月内降低体脂率1%），系统按进度提供阶段性反馈，通过目标感驱动用户坚持使用体脂监测功能。

预期效果： 体脂分析功能使用频率提升至总记录的15%以上，用户对该功能的满意度显著提高，体成分数据质量明显改善。

题号 3.1.5-2 优化方向与解决方案

基于以上数据分析，提出三个核心优化方向及对应解决方案：

优化方向一：基于用户行为模式的个性化场景预测与自动调节

问题背景：

数据显示，用户在不同时段对温度（偏差 ±0.7°C）、湿度（偏差 ±3.5%）和光照（偏差 ±20 lux）有规律性偏好，但当前系统仍依赖用户手动操作触发，无法提前预判并自动匹配用户习惯。

解决方案：

  1. 构建用户行为画像模型 - 收集用户历史操作数据，按时段、星期、季节三个维度建立个性化偏好矩阵。 - 使用 K-Means 聚类或协同过滤算法对用户偏好进行分组，识别相似行为模式。 - 为每位用户生成"环境偏好时间曲线"，涵盖温度、湿度、光照三项关键指标。

  2. 部署预测性场景联动引擎 - 在用户习惯操作时间前 10–15 分钟自动预调节环境参数，实现无感切换。 - 例如：每日 21 时用户通常将光照降至 300 lux 以下，系统可在 20:45 开始渐进调暗，避免突变。 - 引入"确认反馈"机制：首次自动调节后提示用户确认，迭代优化预测精度。

  3. 融合多源感知数据 - 结合位置传感器（用户到家/离家）、日历信息（节假日/工作日）和天气预报 API 动态修正预测模型。 - 节假日模式自动切换，避免工作日模型错误应用于休息日。

预期效果： 减少用户手动操作频次约 40–60%，提升环境舒适度满意率，降低因响应延迟带来的用户干预成本。

优化方向二：系统响应时间优化——本地边缘计算与分级响应架构

问题背景：

当前系统平均响应时间为 3.02 秒，且有 27.5% 的操作响应超过 4 秒。响应时间与环境参数几乎无相关性，说明延迟主要来源于系统架构层面（云端处理往返延迟、协议开销），而非传感器数据本身。

解决方案：

  1. 引入本地边缘计算网关 - 在家庭本地部署边缘计算节点，将常用操作指令（如灯光开关、温度快速调节）的逻辑下沉至本地执行，绕过云端往返，将响应时间目标降至 < 1 秒。 - 本地网关负责处理延迟敏感型操作（响应时间 < 2 秒），云端处理复杂场景联动和机器学习推理（容忍 3–5 秒延迟）。

  2. 建立分级响应优先级机制 - 将操作类型按响应优先级分为三级：

    • 一级（实时）：安防联动、紧急关闭、语音即时控制 → 目标 < 500ms，本地执行。
    • 二级（快速）：常用环境调节（灯光、温度） → 目标 < 1.5 秒，本地 + 边缘执行。
    • 三级（标准）：场景模式切换、定时任务、数据上报 → 目标 < 5 秒，云端执行。
    • 高优先级请求在网络拥塞时自动降级为本地离线执行，保障核心操作不受影响。

  3. 优化通信协议与连接管理 - 对 Wi-Fi 连接的设备推行长连接保活策略，减少每次操作的握手开销。 - 对 Zigbee/Z-Wave 设备实施自适应信道选择，在早晨操作高峰期（响应时间峰值 3.10 秒）自动切换至干扰较少的信道。 - 引入请求队列批处理机制：在 100ms 时间窗内合并来自同一场景的多条指令，减少协议交互次数。

预期效果： 系统平均响应时间从 3.02 秒降至 < 1.5 秒，响应超过 4 秒的操作占比从 27.5% 降至 < 5%，早晨峰值响应时间降低约 50%。

优化方向三：智能节能管理——动态能耗优化与峰谷错峰调度

问题背景：

数据分析显示，高光照场景能耗比低光照高 12.7%，早晨时段能耗比深夜高 13.6%，夏季能耗比春季高 11.2%。整体能耗标准差达 0.54 kWh（占均值的 53%），说明单次操作能耗波动很大，存在显著的节能优化空间。

解决方案：

  1. 自适应照明节能策略 - 基于室外自然光照传感器实时数据，动态补偿室内照明亮度：当室外光线充足时，自动降低人工照明功率，避免过度照明浪费。 - 对高光照使用场景（> 600 lux，占比 44%）设置场景必要性判断：若用户正在休息或无人时，自动将光照降至 200–300 lux 的节能模式，预计可降低照明能耗约 15–20%。 - 推行渐变调光策略（Dimming Curve），替代突然开/关操作，延长 LED 灯具寿命同时减少瞬间功率冲击。

  2. 峰谷错峰调度与电力需求响应 - 对非实时性任务（如地暖预热、空气净化器运行、热水器加热）实施电力峰谷时段调度：将高能耗任务从早晨（6–9 时，能耗峰值 1.097 kWh）移至深夜低谷时段（22–6 时，能耗 0.966 kWh），利用峰谷电价差降低电费成本约 10–15%。 - 接入电网需求响应（DR）信号，在电网高峰时段自动执行节能模式，配合智能电网政策获取用电优惠。

  3. 季节自适应能耗模型 - 针对夏季制冷能耗高（均值 1.056 kWh）的问题，建立温度死区控制策略：将制冷启停温度死区从 ±0.5°C 扩大至 ±1°C，减少压缩机频繁启停次数，降低能耗约 5–8%。 - 冬季利用用户偏好数据（冬季平均温度设定 24.95°C），结合室外气温预报实施预热策略：在用户回家前 30 分钟预先启动暖气，避免用户到家后高功率快速加热造成的峰值能耗。 - 建立月度能耗基线（Baseline）报告，自动对比当月实际能耗与基线，超出阈值（如 +10%）时主动推送节能建议给用户。

预期效果： 综合节能率目标 10–20%，年度总能耗从当前估算约 1020.87 kWh 降至 820–920 kWh，同时提升用户对能源使用的可见性和控制感。

批量处理： - 支持用户一次上传多张图片（最多 20 张），系统以批次方式送入模型推理，充分利用硬件并行计算能力，减少单张处理的平均耗时 - 批处理结果以列表形式展示，用户可逐条查看详情
